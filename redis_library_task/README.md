# 🏛️ Практическая работа: Внедрение Redis-кэширования в онлайн-библиотеку

## Описание задания

Вам дан готовый backend **онлайн-библиотеки** (FastAPI + PostgreSQL) без кэширования.  
Ваша задача — **проанализировать** каждый эндпоинт, **решить** где нужен Redis-кэш и **реализовать** кэширование.

Проект содержит три сущности с **разной «горячестью» данных**:

- **Author** (аналог Category в магазине) — читается часто, пишется почти никогда. Справочник авторов.
- **Book** (аналог Product) — читается очень часто, пишется редко. Основная «горячая» таблица.
- **Borrowing** (аналог Order) — читается средне, пишется постоянно. Каждая выдача/возврат — запись.

## Список всех эндпоинтов

### Authors (авторы)

- `GET /authors/` — список всех авторов. Вызывается на каждой странице (меню, фильтры). Данные меняются почти никогда.
- `GET /authors/{id}` — страница автора. Частота средняя. Данные меняются почти никогда.

### Books (книги)

- `GET /books/` — каталог с пагинацией и фильтром по автору. Частота высокая. Данные меняются редко.
- `GET /books/top-rated` — топ книг по рейтингу. Частота очень высокая (главная страница). Данные меняются редко.
- `GET /books/popular` — самые просматриваемые книги. Частота очень высокая. Данные меняются часто (views_count растёт).
- `GET /books/count` — количество доступных книг. Частота средняя. Данные меняются редко.
- `GET /books/search?q=` — поиск по названию. Частота высокая. Каждый запрос уникален.
- `GET /books/{id}` — карточка книги. Частота очень высокая для популярных книг. Данные меняются редко.
- `POST /books/` — добавить книгу. Частота низкая. Операция записи.
- `PATCH /books/{id}` — обновить книгу. Частота низкая. Операция записи.

### Borrowings (выдачи)

- `POST /borrowings/` — выдать книгу читателю. Частота высокая. Операция записи.
- `POST /borrowings/{id}/return` — вернуть книгу. Частота высокая. Операция записи.
- `GET /borrowings/` — список всех выдач (для библиотекаря). Частота средняя. Данные меняются постоянно.
- `GET /borrowings/{id}` — конкретная выдача. Частота низкая. Данные меняются иногда.

### Часть 2 — Реализация

#### Шаг 1. Инфраструктура Redis

1. Добавьте `redis[hiredis]==5.2.1` в `requirements.txt`
2. Добавьте Redis-контейнер в `docker-compose.yml`:
   ```yaml
   redis:
     image: redis:7-alpine
     container_name: library_redis
     ports:
       - "6380:6379"
     command: redis-server --maxmemory 64mb --maxmemory-policy allkeys-lru
     healthcheck:
       test: ["CMD", "redis-cli", "ping"]
       interval: 5s
       timeout: 3s
       retries: 5
   ```
3. Добавьте `redis_url` в `config.py`
4. Не забудьте добавить `depends_on: redis` для сервиса `app`

#### Шаг 2. Redis-клиент

Создайте файл `app/redis_client.py` — async singleton для подключения к Redis:
- Функция `get_redis()` — создать или вернуть пул подключений
- Функция `close_redis()` — закрыть пул при остановке приложения

#### Шаг 3. Cache Service

Создайте файл `app/cache_service.py` с логикой кэширования:
- Базовые функции: `cache_get`, `cache_set`, `cache_delete`
- TTL с jitter (случайная добавка) — защита от Cache Stampede / Avalanche
- Высокоуровневые функции для каждого кэшируемого эндпоинта
- Функции инвалидации

#### Шаг 4. Подключение Redis в main.py

В `lifespan` приложения добавьте:
- Инициализацию Redis при старте
- Закрытие Redis при остановке

#### Шаг 5. Добавление кэша в роуты

Для каждого эндпоинта, который вы решили кэшировать (из Части 1):
1. Импортируйте нужные функции из `cache_service`
2. В начале обработчика — проверьте кэш (`cache_get`)
3. Если cache hit — верните данные из кэша (без обращения к PostgreSQL)
4. Если cache miss — выполните запрос к БД, затем сохраните в кэш (`cache_set`)
5. Для операций записи (PATCH) — добавьте инвалидацию связанных ключей

## Подсказки

- Паттерн **Cache-Aside**: `проверь кэш → если miss → сходи в БД → положи в кэш`
- **TTL jitter**: добавляйте случайное число к TTL, чтобы ключи не протухали одновременно
- **Инвалидация**: при записи удаляйте связанные ключи кэша
- Кэшируйте **только** то, что: (а) читается часто, (б) одинаково для всех, (в) меняется редко
- Для удаления ключей по шаблону используйте `SCAN` + `DELETE` (не `KEYS` — он блокирует Redis)

---

## Частые ошибки

1. **Кэшировать всё подряд** — бессмысленно, Redis тоже ресурс
2. **Забыть про инвалидацию** — пользователи видят устаревшие данные
3. **Одинаковый TTL для всех** — данные имеют разную частоту обновления  
4. **Кэшировать поиск с тысячами комбинаций** — низкий hit rate, много памяти
5. **Кэшировать операции записи** — POST/PATCH/DELETE не кэшируют
6. **Не обрабатывать падение Redis** — приложение должно работать и без кэша
