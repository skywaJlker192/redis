# Практическая работа: Redis-кэширование в backend-приложении

## Контекст

Вы разработчик в небольшой команде. Вам передали работающий backend онлайн-библиотеки на FastAPI + PostgreSQL. Приложение справляется с нагрузкой, но с ростом аудитории база данных начинает «задыхаться» — одни и те же данные запрашиваются сотни раз в секунду.

Ваша задача — внедрить Redis-кэширование, предварительно разобравшись, какие именно данные стоит кэшировать, а какие — нет.

---

## Домен: онлайн-библиотека

Приложение управляет тремя сущностями:

- **Author** — справочник авторов. Создаётся редко, читается на каждой странице.
- **Book** — каталог книг. Пишется редко, читается очень часто.
- **Borrowing** — факт выдачи книги читателю. Создаётся при каждой выдаче и возврате.

## Эндпоинты приложения

### Authors

- `GET /authors/` — список всех авторов
- `GET /authors/{id}` — автор по ID

### Books

- `GET /books/` — каталог с пагинацией, фильтр по `author_id`
- `GET /books/top-rated` — топ книг по рейтингу
- `GET /books/popular` — топ книг по количеству просмотров
- `GET /books/count` — количество доступных книг
- `GET /books/search?q=` — поиск книг по названию
- `GET /books/{id}` — карточка книги
- `POST /books/` — добавить книгу
- `PATCH /books/{id}` — обновить книгу

### Borrowings

- `POST /borrowings/` — выдать книгу читателю
- `POST /borrowings/{id}/return` — вернуть книгу
- `GET /borrowings/` — список выдач
- `GET /borrowings/{id}` — выдача по ID

### Часть 2 — Реализация

Реализуйте кэширование в папке `task/`. Не меняйте бизнес-логику — только добавляйте слой кэша.

#### Шаг 1. Зависимости и инфраструктура

Добавьте в `requirements.txt`:

```
redis[hiredis]==5.2.1
```

Добавьте Redis-контейнер в `docker-compose.yml` и `redis_url` в `config.py`.
Сервис `app` должен зависеть от готовности Redis (`depends_on` с `condition: service_healthy`).

#### Шаг 2. Создайте `app/redis_client.py`

Файл должен содержать:

- асинхронный singleton-клиент Redis
- функцию `get_redis()` — возвращает подключение
- функцию `close_redis()` — закрывает пул при остановке приложения

#### Шаг 3. Создайте `app/cache_service.py`

Файл должен содержать всю логику кэширования:

- базовые операции `cache_get`, `cache_set`, `cache_delete`
- функцию для TTL с jitter (добавить случайное число к базовому TTL)
- высокоуровневые функции `get_cached_*` / `set_cached_*` для каждого кэшируемого эндпоинта
- функции инвалидации `invalidate_*`

#### Шаг 4. Подключите Redis в `app/main.py`

В `lifespan` добавьте инициализацию Redis при старте и закрытие при остановке.

#### Шаг 5. Добавьте кэш в роуты

Для каждого эндпоинта из Части 1, который вы решили кэшировать:

- проверьте кэш в начале обработчика
- при cache hit — верните данные из Redis без обращения к БД
- при cache miss — выполните запрос к БД, сохраните результат в Redis
- для `PATCH`-эндпоинтов — добавьте инвалидацию после успешного обновления
- для `POST /borrowings/` и `POST /borrowings/{id}/return` — подумайте, нужна ли инвалидация

#### Шаг 6. Запустите бенчмарк

Запустите `task/` на порту 8000, затем `solution/` на порту 8001, затем:

```bash
cd ..
pip install aiohttp
python benchmark.py
```

## Справочная информация

Паттерн **Cache-Aside**:

```python
cached = await cache_get(key)
if cached:
    return cached
data = await db_query()
await cache_set(key, data, ttl)
return data
```

**TTL jitter** — случайная добавка к TTL предотвращает одновременное протухание множества ключей и массовый удар по БД:

```python
ttl = base_ttl + random.randint(0, jitter)
```

**Инвалидация** — при изменении данных удаляйте связанные ключи кэша. Для удаления ключей по шаблону используйте `SCAN` + `DELETE`, не `KEYS` (он блокирует Redis).
