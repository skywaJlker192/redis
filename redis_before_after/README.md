# Redis Before & After

## Идея

Два одинаковых проекта — интернет-магазин (FastAPI + PostgreSQL). Первый работает без кеша, второй — с Redis. Можно сравнить файлы и увидеть, что именно меняется при внедрении Redis в существующий backend, какие эндпоинты получают кеш и почему, а какие — нет.

Redis ставится не на всё подряд, а только на горячие данные: категории (меняются редко), популярные товары (одинаковые для всех) и карточку товара (частый запрос). Заказы, пагинация и операции записи не кешируются.

## Запуск

```bash
# Без кеша (port 8000)
cd 01_no_cache
docker compose up -d --build

# С Redis (port 8001)
cd ../02_with_redis
docker compose up -d --build

# Бенчмарк (сравнение скорости)
cd ..
pip install aiohttp
python benchmark_v2.py
```

## Что добавилось во втором проекте

Два новых файла: `redis_client.py` (подключение к Redis) и `cache_service.py` (логика кеширования — Cache-Aside, TTL с jitter, инвалидация).

В существующих файлах минимальные изменения: в `config.py` добавлена строка `redis_url`, в `main.py` — подключение и отключение Redis при старте, в роутах `categories.py` и `products.py` — обёртка «проверь кеш → если пусто, сходи в БД → положи в кеш». При обновлении товара через `PATCH` кеш инвалидируется. В `docker-compose.yml` добавлен Redis-контейнер.

Модели, схемы, seed-данные и роуты заказов не изменились вообще.

## Результаты

На чистой задержке (один пользователь, 100 запросов подряд) кешируемые эндпоинты с Redis отвечают примерно в 2 раза быстрее: категории ~0.47 мс вместо ~0.87 мс, карточка товара ~0.45 мс вместо ~0.98 мс. На хвостах (P99) разница ещё заметнее — до 5-6 раз.

Под нагрузкой (50-100 параллельных пользователей) Redis даёт +10-12% к пропускной способности и снижает медианную задержку в 1.5 раза.

На наших 24 товарах PostgreSQL и сам отвечает быстро, поэтому разница «всего» ×2. В реальных проектах с тяжёлыми JOIN, полнотекстовым поиском или миллионами строк Redis отвечает за те же ~0.5 мс, а БД за 50-2000 мс, и выигрыш составляет ×100-4000.
